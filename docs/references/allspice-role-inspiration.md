# AllSpice Role Inspiration

## Purpose

This note preserves the job-posting context that inspired BOMFlow without making the Discovery narrative company-specific.

BOMFlow should stand on its own as a credible hardware workflow automation prototype. The AllSpice context is useful for alignment, scoping, and portfolio evaluation, but the product problem should remain framed around hardware teams and their review workflows.

## Signals To Preserve

- Hardware workflow automation.
- Actions-style scripts that can run inside or alongside a design review process.
- Integrations and structured handoffs between engineering systems.
- Forward-deployed engineering judgment: start from a real customer workflow, build the smallest useful automation, and leave a clean path toward reuse.
- Example project inspiration: a carbon emissions report for parts in a design.

## How This Should Shape BOMFlow

- Start with a single-BOM carbon report that is useful without future roadmap items.
- Keep data mocked and methodology limits explicit.
- Design outputs as review artifacts: clear enough for engineers, managers, and sustainability/compliance reviewers to act on.
- Leave architectural room for later deltas, design review automation, and PLM/manufacturing sync.

## How This Should Not Shape BOMFlow

- Do not make the problem statement about getting a job.
- Do not make AllSpice the primary user.
- Do not overfit names, APIs, or implementation details to a company-specific platform before the user workflow is clear.
- Do not imply production-grade emissions science.
